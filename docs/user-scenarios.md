# Cowork 用户使用场景文档

## 一、目标用户画像

### 1.1 典型用户

**个人开发者 / 独立创业者**

cowork 的典型用户是个人开发者或独立创业者，他们有以下特点：

| 特点 | 描述 |
|------|------|
| **技术背景** | 熟悉编程，习惯使用命令行工具 |
| **工作模式** | 同时处理多个项目或任务，需要并行执行能力 |
| **资源限制** | 个人资源有限，希望最大化利用现有设备 |
| **效率导向** | 追求自动化，希望减少重复性工作 |
| **AI 原生** | 习惯使用 AI 工具辅助工作 |

### 1.2 用户需求

1. **并行处理**：同时跑多个耗时任务（如代码生成、数据分析、测试），不想一个一个等待
2. **灵活调度**：不同任务需要不同能力（如需要 GPU、需要 Docker 环境），希望自动匹配
3. **实时监控**：随时了解任务进度，不想频繁刷新页面
4. **AI 辅助**：日常工作中需要 AI 对话支持（代码问答、文档生成等）
5. **个性化界面**：按照自己的工作习惯定制 Dashboard 布局

### 1.3 用户故事

> "我是一名独立开发者，同时维护 3 个项目。每天早上，我会启动几个后台任务（代码检查、测试运行、文档生成），然后通过 Agent Chat 让 AI 帮我分析代码。我希望有一个统一的地方能看到所有任务的进度，并且不需要在多个工具之间切换。"

## 二、核心使用场景

### 场景一：批量代码分析

#### 1. 场景名称
**多项目并行代码审查**

#### 2. 用户诉求
用户同时维护多个代码仓库，希望并行运行代码分析任务，快速获得质量报告。

#### 3. 操作流程

```
1. 用户打开 cowork Dashboard
2. 进入「编辑模式」，添加 3 个「任务队列」Widget（每个项目一个）
3. 点击「添加 Widget」，选择「统计面板」查看整体进度
4. 通过 API 或界面提交 3 个分析任务：
   - 任务 A：检查 project-alpha 的代码规范
   - 任务 B：运行 project-beta 的单元测试
   - 任务 C：生成 project-gamma 的 API 文档
5. Dashboard 实时显示任务进度：
   - 任务 A: 45% (正在检查 src/ 目录)
   - 任务 B: 80% (测试通过 156/200)
   - 任务 C: 30% (正在解析注释)
6. 任务完成后，收到通知，点击查看详细报告
```

#### 4. 角色配合

| 角色 | 动作 | 说明 |
|------|------|------|
| **Coordinator** | 接收任务创建请求 | 记录任务信息，状态设为 `pending` |
| **Coordinator** | 调度任务分配 | 查找匹配标签的空闲 Worker，分配任务 |
| **Worker 1** | 执行任务 A | 标签 `dev,lint`，处理代码规范检查 |
| **Worker 2** | 执行任务 B | 标签 `dev,test`，运行单元测试 |
| **Worker 3** | 执行任务 C | 标签 `dev,doc`，生成文档 |
| **Coordinator** | 收集心跳和进度 | 更新数据库，通过 WebSocket 推送给前端 |
| **Dashboard** | 实时更新界面 | 显示进度条和任务状态 |
| **Notification** | 推送完成通知 | 任务完成后触发通知 Widget |

#### 5. 数据流转

```
┌──────────┐    创建任务     ┌──────────────┐
│  用户    │ ───────────────▶│ Coordinator  │
└──────────┘                 └──────┬───────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
               ┌─────────┐    ┌─────────┐    ┌─────────┐
               │Worker 1 │    │Worker 2 │    │Worker 3 │
               │(lint)   │    │(test)   │    │(doc)    │
               └────┬────┘    └────┬────┘    └────┬────┘
                    │               │               │
                    │   心跳+进度   │               │
                    └───────────────┼───────────────┘
                                    ▼
                            ┌──────────────┐
                            │ Coordinator  │
                            │ (WebSocket)  │
                            └──────┬───────┘
                                   │
                                   ▼
                            ┌──────────────┐
                            │  Dashboard   │
                            │  (实时更新)  │
                            └──────────────┘
```

### 场景二：AI 辅助编程

#### 1. 场景名称
**智能代码问答与生成**

#### 2. 用户诉求
用户在开发过程中遇到问题，希望通过 AI Agent 获得帮助，如代码解释、Bug 分析、代码生成等。

#### 3. 操作流程

```
1. 用户在 Dashboard 添加「Agent Chat」Widget
2. 选择模型（如 GPT-4、Claude、GLM-4）
3. 输入问题："帮我分析这段代码的性能瓶颈"
4. 可选：上传相关文件作为上下文
5. Agent 开始流式回复：
   "让我看看这段代码...
    主要瓶颈在于循环中的数据库查询...
    建议修改为批量查询..."
6. 用户追问："能给出修改后的代码吗？"
7. Agent 继续对话，生成代码建议
8. 用户可以将对话导出或关联到某个任务
```

#### 4. 角色配合

| 角色 | 动作 | 说明 |
|------|------|------|
| **Dashboard** | 创建 Agent 会话 | 调用 `/api/agent/sessions`，获取 session_id |
| **Coordinator** | 路由请求 | 根据模型选择路由到对应的 LLM API |
| **LLM Provider** | 生成回复 | OpenAI / Anthropic / 智谱等返回流式响应 |
| **Coordinator** | 流式推送 | 通过 SSE 或 WebSocket 推送每个 chunk |
| **Agent Chat Widget** | 实时渲染 | 逐字显示 AI 回复，支持 Markdown |
| **Coordinator** | 保存历史 | 会话消息持久化到 SQLite |

#### 5. 数据流转

```
┌──────────┐    发送消息     ┌──────────────┐
│  用户    │ ───────────────▶│ Coordinator  │
└──────────┘                 └──────┬───────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
              ┌──────────┐  ┌──────────┐  ┌──────────┐
              │  OpenAI  │  │ Anthropic│  │  智谱    │
              │  (GPT-4) │  │ (Claude) │  │  (GLM-4) │
              └────┬─────┘  └────┬─────┘  └────┬─────┘
                   │             │             │
                   └─────────────┼─────────────┘
                                 │ 流式响应
                                 ▼
                         ┌──────────────┐
                         │ Coordinator  │
                         │  (SSE 推送)  │
                         └──────┬───────┘
                                │
                                ▼
                         ┌──────────────┐
                         │ Agent Chat   │
                         │   Widget     │
                         └──────────────┘
```

### 场景三：Docker 容器任务

#### 1. 场景名称
**隔离环境中的安全执行**

#### 2. 用户诉求
用户需要运行一些不可信或需要特定环境的任务，希望在隔离的 Docker 容器中执行，确保主机安全。

#### 3. 操作流程

```
1. 用户提交任务，指定需要 Docker 环境：
   - type: "sandbox-analysis"
   - required_tags: ["docker", "isolated"]
   - input: { "script_url": "..." }
2. 系统自动匹配支持 Docker 的 Worker
3. Worker 为任务创建独立容器：
   - 限制 CPU: 1 核
   - 限制内存: 512MB
   - 禁止网络访问（可选）
4. 任务在容器内执行
5. 执行完成后：
   - 收集输出文件
   - 销毁容器
   - 上报结果
6. 用户下载输出文件
```

#### 4. 角色配合

| 角色 | 动作 | 说明 |
|------|------|------|
| **Coordinator** | 标签匹配 | 筛选支持 `docker` 标签的 Worker |
| **Worker (Docker)** | 创建容器 | `docker run --cpus=1 --memory=512m ...` |
| **Worker (Docker)** | 执行任务 | 在容器内运行脚本 |
| **Worker (Docker)** | 收集结果 | 从容器中提取输出文件 |
| **Coordinator** | 存储文件 | 任务文件保存到本地或 S3 |
| **Dashboard** | 显示结果 | 提供文件下载链接 |

#### 5. 数据流转

```
┌──────────┐    提交任务     ┌──────────────┐
│  用户    │ ───────────────▶│ Coordinator  │
└──────────┘                 └──────┬───────┘
                                   │ 标签匹配
                                   ▼
                            ┌──────────────┐
                            │ Docker Worker│
                            └──────┬───────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
              ┌──────────┐  ┌──────────┐  ┌──────────┐
              │ Container│  │ Container│  │ Container│
              │ (Task A) │  │ (Task B) │  │ (Task C) │
              └────┬─────┘  └────┬─────┘  └────┬─────┘
                   │             │             │
                   └─────────────┼─────────────┘
                                 │ 结果收集
                                 ▼
                         ┌──────────────┐
                         │   文件存储   │
                         └──────┬───────┘
                                │
                                ▼
                         ┌──────────────┐
                         │   Dashboard  │
                         │  (下载文件)  │
                         └──────────────┘
```

### 场景四：个性化工作台

#### 1. 场景名称
**自定义 Dashboard 布局**

#### 2. 用户诉求
用户希望根据自己的工作习惯，定制 Dashboard 的布局和内容，不同时间关注不同的信息。

#### 3. 操作流程

```
1. 用户进入「编辑模式」
2. 拖拽 Widget 调整位置和大小：
   - 左上角：任务队列（大）
   - 右上角：Agent Chat（大）
   - 左下角：Worker 状态（中）
   - 右下角：通知中心（小）
3. 添加新的 Widget：
   - 点击「添加 Widget」按钮
   - 从 Widget Store 中选择
   - 配置 Widget 参数（如刷新间隔、显示数量）
4. 删除不需要的 Widget
5. 布局自动保存，刷新页面后恢复
```

#### 4. 角色配合

| 角色 | 动作 | 说明 |

| **Dashboard** | 拖拽交互 | react-grid-layout 处理拖拽逻辑 |

| **localStorage** | 本地缓存 | 立即保存到浏览器本地存储 |
| **Coordinator** | 云端同步 | 调用 `/api/user/layout` 同步到服务器 |
| **Coordinator** | 持久化 | 保存到 SQLite `user_layouts` 表 |

### 场景五：Worker 集群管理

#### 1. 场景名称
**多 Worker 状态监控与任务分发**

#### 2. 用户诉求
用户在不同设备上部署了多个 Worker，希望统一监控状态、管理任务分发。

#### 3. 操作流程

```
1. 用户在多台设备启动 Worker：
   - 笔记本：worker-laptop --tags=dev,coding --model=gpt-4
   - 云服务器：worker-cloud --tags=docker,gpu --model=glm-4
   - 树莓派：worker-pi --tags=iot,monitor --model=claude

2. 各 Worker 向 Coordinator 注册并开始心跳

3. 用户在 Dashboard 添加「Worker 状态」Widget：
   - 显示所有在线 Worker
   - 实时状态：idle/busy/offline
   - 资源使用：CPU、内存
   - 当前任务数

4. 提交任务时指定标签偏好：
   - 任务 A (需要 GPU)：required_tags=["gpu"]
   - 任务 B (普通代码)：required_tags=["coding"]

5. 系统自动匹配最合适的 Worker

6. Worker 下线时，自动重新分配其任务
```

#### 4. 角色配合

| 角色 | 动作 | 说明 |
|------|------|------|
| **Worker** | 注册 | 启动时调用 `/api/workers/register` |
| **Worker** | 心跳 | 每 5 秒发送心跳，包含状态和进度 |
| **Coordinator** | 状态管理 | 更新 Worker 状态，检测超时 |
| **Coordinator** | 调度匹配 | 根据标签和负载分配任务 |
| **Dashboard** | 可视化 | WebSocket 实时更新 Worker 状态 |
| **Coordinator** | 故障转移 | Worker 离线后重新分配任务 |

#### 5. 数据流转

```
Worker 启动流程：

┌─────────────┐    注册请求    ┌──────────────┐
│   Worker    │ ──────────────▶│ Coordinator  │
│  (启动中)   │                │              │
└─────────────┘                └──────┬───────┘
                                      │ 分配 ID
                                      ▼
                               ┌──────────────┐
                               │   Worker ID  │
                               │  + 配置信息  │
                               └──────────────┘

心跳循环（每 5 秒）：

┌─────────────┐    心跳+状态    ┌──────────────┐
│   Worker    │ ──────────────▶│ Coordinator  │
│  (运行中)   │ ◀────────────── │              │
└─────────────┘   新任务分配    └──────┬───────┘
                                      │
                                      ▼
                               ┌──────────────┐
                               │  Dashboard   │
                               │ (实时更新)   │
                               └──────────────┘

故障转移：

┌─────────────┐    心跳超时    ┌──────────────┐
│   Worker    │ ──────────────▶│ Coordinator  │
│  (离线)     │   (30秒无响应) │              │
└─────────────┘                └──────┬───────┐
                                      │       │
                              标记离线│       │重新分配
                                      ▼       ▼
                               ┌──────────────┐
                               │ 其他 Worker  │
                               │ (接管任务)   │
                               └──────────────┘
```

### 3.1 Coordinator（协调者）

Coordinator 是整个系统的核心大脑，负责统筹协调。

#### 核心职责

| 职责 | 说明 |
|------|------|
| **API 网关** | 提供统一的 REST API 入口，处理所有 HTTP 请求 |
| **任务调度** | 接收任务请求，匹配 Worker，实现负载均衡 |
| **Worker 管理** | 注册、心跳、状态监控、故障检测 |
| **实时推送** | WebSocket Hub，向客户端推送任务状态、通知等 |
| **Agent 路由** | 将 AI 对话请求路由到不同的 LLM Provider |
| **数据存储** | 任务、Worker、会话、布局等数据的持久化 |

### 3.2 Worker（执行者）

Worker 是实际干活的节点，负责执行具体任务。

#### 核心职责

| 职责 | 说明 |
|------|------|
| **任务执行** | 接收并执行分配的任务 |
| **进度上报** | 定期向 Coordinator 汇报任务进度 |
| **心跳维持** | 保持与 Coordinator 的连接，报告存活状态 |
| **工作目录管理** | 为每个任务创建隔离的工作空间 |
| **结果输出** | 生成输出文件，上报执行结果 |

#### 核心能力

```

│  │ • 模型配置  │  │ • 队列管理  │                      │

│  └─────────────┘  └─────────────┘                      │
```

#### Worker 类型

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **本地 Worker** | 直接在主机运行，无隔离 | 受信任任务、高性能需求 |
| **Docker Worker** | 每任务一个容器，完全隔离 | 不可信代码、多环境需求 |
| **云端 Worker** | 部署在云服务器，可弹性扩展 | 大规模任务、需要 GPU |

### 3.4 Agent Chat（AI 对话能力）

Agent Chat 提供 AI 对话能力，是用户获取 AI 辅助的主要入口。

#### 核心职责

| 职责 | 说明 |
|------|------|
| **多模型支持** | 支持 GPT-4、Claude、GLM-4 等多种模型 |
| **流式响应** | 实时显示 AI 回复，提升用户体验 |
| **上下文管理** | 管理对话历史，支持多轮对话 |
| **文件关联** | 支持上传文件作为上下文 |

#### 支持的模型

| Provider | 模型 | 特点 |
|----------|------|------|
| OpenAI | GPT-4 | 通用能力强，适合复杂任务 |
| Anthropic | Claude 3 | 安全性高，长文本处理 |
| 智谱 AI | GLM-4 | 中文优化，性价比高 |

## 五、总结

### 5.1 核心价值

| 价值点 | 说明 |
|--------|------|
| **并行执行** | 多 Worker 同时处理任务，大幅提升效率 |
| **灵活调度** | 基于标签的任务分发，自动匹配最合适的 Worker |
| **实时监控** | WebSocket 实时推送，随时掌握任务状态 |
| **个性定制** | 可拖拽 Dashboard，按需配置工作台 |
| **AI 原生** | 多模型 Agent Chat，无缝集成 AI 能力 |

### 5.2 适用场景

- 多项目并行开发
- 批量代码分析和测试
- AI 辅助编程
- 容器化任务隔离执行
- 个人工作流自动化

### 5.3 快速上手

```bash
# 1. 启动 Coordinator
go run ./cmd/coordinator

# 2. 启动 Worker（另一个终端）
go run ./cmd/worker --name=worker-1 --tags=dev,coding --model=gpt-4

# 3. 启动前端
cd web && npm run dev

# 4. 打开浏览器
open http://localhost:3000
```
